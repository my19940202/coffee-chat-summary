# NewBee 香港蓝领用工平台 — 开发评估摘要

> 本文档基于飞书多维表格（邦芒确认优先级）及产品 PRD 文档整理，供开发伙伴和产品经理评估工作量。

---

## 一、项目现状与背景

### 1.1 产品定位

NewBee 是面向**香港蓝领用工市场**的数字化平台，核心解决蓝领求职、雇主招工、日结用工、考勤结算等全链路问题。产品参考国内"店伙伴"的成熟业务模型，针对香港市场进行本地化适配。

### 1.2 已完成情况

项目采用 Monorepo 架构（pnpm workspace），技术栈为 **NestJS + Vue 3 + Ant Design Vue + TypeORM + PostgreSQL**。

**已上线模块（5 个前端应用 + 1 个后端服务）：**

| 应用 | 端口 | 说明 |
|------|------|------|
| Server (NestJS) | 3000 | 后端 API，42 个实体、18 个 Controller |
| Platform | 5173 | 平台管理后台 |
| Merchant | 5174 | 商户 H5 端 |
| Employee | 5175 | 雇员 H5 端 |
| Merchant-Admin | 5176 | 商户管理后台 |

**已实现的核心能力：**

- 账号体系：手机号注册登录、OTP 验证、JWT 鉴权
- 岗位模块：岗位发布/列表/详情/搜索/筛选
- 报名模块：雇员报名、雇主处理报名
- 排班模块：排班创建/模板/日历视图
- 考勤模块：GPS 打卡 + 水印照片、考勤确认→发薪
- 结算模块：钱包/提现/FPS 支付、商户账单生成→确认→支付→逾期跟踪
- 团队管理：邀请链接（24h 有效期）、自定义角色 + 人员范围（全员/按分组）
- 雇员分组：雇员多分组管理
- 商户授信：L1-VIP 客户等级、信用额度调整（等级额度/指定额度双模式）
- 工作码：WorkerCode 扫码体系
- 保险模块：创建/激活/理赔/过期
- i18n 国际化框架（繁中 + 英文待补充翻译）
- 代发薪：Excel 上传校验、批量代发

### 1.3 业务背景

邦芒（合作方）已基于国内"店伙伴"产品梳理了 9 大核心业务场景，并确认了各功能模块的优先级（P0/P1/P2）。当前需基于此优先级规划分期开发。

**9 大业务场景：**

1. 标准日结 — 单班次日结岗位
2. 多班次聚合 — 多班次同岗、同工不同酬
3. 多地点配送 — 配送截图 OCR + 等候补贴
4. 点位采集 — 地图点位 GPS + 照片打卡
5. 远程音频 — AI 音频审核
6. 代发薪（含对账）— 客户上传薪资表 + 双考勤对账
7. 纯代发薪 — 直接导入薪资表批量代发
8. 大型活动 — 驻场管理 + 保证金 + WhatsApp 群组
9. 长工外包 — 平台代理账号代客户发布

## 二、分期开发规划

### 第一期（P0）— 核心必做

> 目标：跑通"商户准入→岗位发布→报名→排班→考勤→结算"核心闭环

| 模块 | 子功能 | 功能说明 | 已有基础 | 待开发工作 | 预估(人天) |
|------|--------|----------|----------|-----------|-----------|
| 商户准入与授信 | 商户认证审核 | BR+HKID+公司信息→平台审核→签署协议→开通账号 | Merchant 有 verificationStatus 字段，MerchantAgreement 实体已有 | 增加 BR 商业登记证 OCR 识别、HKID 正反面上传 | 8 |
| 商户准入与授信 | 商户分级与授信 | L1-VIP 客户等级自动晋级、信用额度管理、垫资上限校验 | Merchant 有 customerLevel/creditMode/maxAdvanceAmount 字段 | 增加自动晋级逻辑和垫资校验 | 5 |
| 报名模块 | 候选补位机制 | 正式岗缺勤→候选补位邀约→30 分钟确认→超时失效 | Application 有 candidate 类型，但无补位流程 | 新增 CandidateSlot 实体和补位流程 | 10 |
| 考勤模块 | GPS 打卡 | GPS 定位打卡 + 水印照片 | 已实现 GPS 打卡 + 水印照片 | 照片比对，确保打卡的是注册账号本人 | 1 |
| 考勤模块 | 工时确认 | 考勤确认→发薪 | 已实现 confirm-hours API | 发薪应为导出发薪表格，现阶段发薪仍需系统外落地 | 1 |
| 结算模块 | 钱包和提现 | 雇员钱包、提现、FPS 支付 | 已实现 Wallet+Withdrawal 完整流程 | 增加 FPS 快捷支付 | 3 |
| 结算模块 | 商户账单 | 生成账单→确认→支付→逾期跟踪 | 已实现 MerchantBill 完整流程 | 核心难点：根据雇主不同勾选项自动生成不同价格，让客户觉得高性价比 | 1 |
| 排班模块 | 排班管理 | 排班创建/模板/日历 | 已实现 Schedule+ScheduleTemplate 完整流程 | 雇主排班界面 + 雇员认领（报名）界面 | 1 |
| 消息模块 | 站内通知 | 站内消息通知系统 | 已实现 Notification 系统 | 无需调整 | 1 |
| 核身与工作码 | 冒用检测 | 扫码时人脸二次核验、顶替行为信用分扣分 | WorkCodeScan 有 result(impostor_suspected)/faceVerifyPassed 字段 | 增加人脸比对 API 对接和信用分扣分逻辑 | 10 |
| 消息模块 | WhatsApp 触达 | WhatsApp Business API 消息推送 | 无此功能 | 对接 WhatsApp Business API | 10 |
| 消息模块 | 多渠道触达优先级 | 站内→WhatsApp→PUSH→SMS | 有 PushFallbackLog 降级日志 | 调整触达优先级链路 | 5 |
| 人才模块 | 信用体系 | 针对员工信用评分，放鸽子等 | 无 | 需设计信用评分模型 | 待评估 |

**P0 期小计：约 56 人天 + 信用体系待评估**

## 四、关键风险与待讨论项

| # | 议题 | 说明 | 影响范围 |
|---|------|------|---------|
| 1 | WhatsApp Business API 对接 | P0 的触达渠道 + P1 的群组功能均依赖，但 API 申请和对接方案尚未确定 | P0/P1 |
| 2 | 人脸识别服务选型 | 冒用检测（P0）需人脸比对 API，有成本（几分到几毛/次），需评估 ROI | P0 |
| 3 | HKID 实名认证 | 香港身份证验证需对接第三方服务，地址证明标准待确认 | P1 |
| 4 | OCR 服务选型 | 配送截图 OCR（P1）需对接 OCR 服务，3 次失败降级逻辑需设计 | P1 |
| 5 | 信用体系设计 | P0 优先级但无现有基础，评分维度（出工量/考勤准点/爽约/雇主评分）需细化 | P0 |
| 6 | 挂靠协议法务 | 雇员侧挂靠协议文案需邦芒准备，可能调整为信息/数据使用授权 | P1 |
| 7 | i18n 翻译质量 | 繁中+英文翻译质量直接影响用户体验，AI 翻译可能不够准确 | P1 |
| 8 | 商户账单定价算法 | P0 核心难点：根据雇主勾选项自动生成价格，需产品侧定义定价规则 | P0 |
| 9 | 代发薪对账流程 | 客户数据 vs 平台数据的对账逻辑需细化 | P1 |
