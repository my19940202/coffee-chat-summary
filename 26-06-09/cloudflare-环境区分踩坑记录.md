# Cloudflare 环境区分踩坑记录

> 日期：2026-06-09  
> 背景：给项目配二级域名、区分开发/正式环境时，在 Cloudflare 上踩了两个坑。

---

## 坑一：二级域名配好了，流量还是跳到别的域名

### 现象

xx.a.com

给项目新配了二级域名，访问却发现还是转发到了**另外一个域名**，怎么改 DNS 都不对，非常诡异。

### 根因

问题不在 DNS，而在 **Cloudflare Rules**。

旧项目里曾经配过一条 **全部转发（catch-all redirect）** 规则。我只盯着 DNS 记录看，完全没往下翻，直到翻到域名配置**最下面的 Rules** 才找到这条规则——它优先级高于 DNS，直接把流量转走了。

### 教训


| 排查顺序         | 说明                                    |
| ------------ | ------------------------------------- |
| 1. DNS 记录    | 先看 A/CNAME 是否指向正确                     |
| 2. **Rules** | 检查 Redirect Rules、Page Rules 等是否有全局转发 |
| 3. Worker 路由 | 确认自定义域名是否绑到了正确的 Worker                |


> **一句话**：DNS 对了不代表流量就到了你的服务，Rules 里的转发规则可以「截胡」。

---

## 坑二：开发环境 vs 正式环境怎么区分

### 需求背景

项目管理员要求把**开发环境**和**正式上线环境**分开：

- **正式环境**：二级域名，例如 `api.example.com`
- **开发/测试环境**：加 `prev` 前缀，例如 `prev-api.example.com`

### 错误尝试一：一个 Worker + wrangler.toml 多环境

参照 AI 建议，在 `wrangler.toml` 里配了两个 env：

```toml
[env.production]
# ...

[env.dev]
# ...
```

**结论：不行。**

一个 Worker 项目本质上只部署一份代码到一处。用 env 切换的方式，没法稳定实现「两个分支、两个域名、两套独立部署」的需求。

### 正确方案：建两个 Cloudflare Worker


| Worker       | 绑定分支              | 绑定域名           | 用途      |
| ------------ | ----------------- | -------------- | ------- |
| Worker A（正式） | `main` / `master` | 正式二级域名         | 生产环境    |
| Worker B（测试） | `test`（或其他开发分支）   | `prev-` 前缀二级域名 | 开发/测试环境 |


开发流程：在 `test` 分支开发 → 验证通过 → 合并到 `main`/`master` → 正式环境自动更新。

---

### 踩坑过程（测试 Worker 折腾了很久）

#### 阶段 1：部署的永远是 main 分支代码

新建了测试 Worker，绑定 `test` 分支和 `prev-` 域名，但部署后代码一直不是最新的，**永远是 main 分支的内容**。

**原因**：忽略了 Cloudflare 构建配置里的 **「非生产分支构建」** 命令。只改了生产分支的构建，非主干分支走的还是默认流程。

#### 阶段 2：预览 URL 对了，自定义域名还是 Hello World

借助 AI 分析后，手动改了 Cloudflare 上的构建命令。这次部署确实生效了——但生效在 `**{commit-hash}.worker.dev`** 这种临时预览 URL 上，每次部署 hash 还不一样。

Meanwhile，绑定在 `test` 分支上的 `**prev-` 自定义域名** 访问过去一直是 `**Hello, World`**（Worker 初始化后的默认内容），死活更新不过去。

#### 阶段 3：根因 —— `upload` ≠ `deploy`

最终定位到 **「非生产分支构建」** 的命令写错了：


| 命令                | 实际效果                                      |
| ----------------- | ----------------------------------------- |
| `wrangler upload` | 只把代码**上传**到 Cloudflare，**不会**把自定义域名的流量切过来 |
| `wrangler deploy` | 真正**部署**，自定义域名才会指向最新代码                    |


之前非生产分支构建写的是 `upload`，所以：

- 预览 URL（`*.worker.dev`）能看到新代码 ✅
- 自定义域名还指着旧版本 / 默认 Hello World ❌

### 最终修复

把测试 Worker 的 **「生产分支构建」** 和 **「非生产分支构建」** 写成**同样的命令**：

```bash
wrangler deploy
```

改完之后：

- `test` 分支每次 push 都触发 CI/CD
- 代码真正部署到 `prev-` 测试域名
- 正式 Worker 继续走 `main`/`master`，互不影响

---

## 核心认知总结

### 1. Cloudflare 的「Preview」≠ 你想要的「测试环境」

非主干分支默认会生成带 commit hash 的临时 URL（`xxx.workers.dev`），这适合**临时预览**，不适合作为**固定测试域名**。

如果你需要稳定的 `prev-xxx.example.com` 测试站，要么：

- **建独立 Worker**（本次采用的方案），或
- 确保非生产分支构建也用 `deploy`，并正确绑定自定义域名

### 2. 一个 Worker 搞不定两套环境

`wrangler.toml` 的 `[env.xxx]` 适合同一套代码切换配置（比如 dev/staging 变量不同），不适合「两个分支、两个域名、独立部署流水线」的场景。**两个环境 = 两个 Worker**，最简单也最不容易踩坑。

### 3. 构建命令一定要分清 upload 和 deploy

```
upload  → 上传代码，域名流量不变
deploy  → 部署上线，域名指向新代码
```

非生产分支如果只配了 `upload`，就会出现「预览 URL 是最新的，自定义域名还是老的」这种迷惑现象。

### 4. 排查域名问题别只看 DNS

```
DNS 记录 → Rules 转发规则 → Worker 路由绑定 → 构建/部署命令
```

四个环节，任何一个断了都会表现为「域名访问不对」。

---

## 推荐检查清单（给未来的自己）

- [ ] 新域名访问异常时，先查 **Rules** 有没有全局转发
- [ ] 开发/正式要分开 → **两个 Worker**，不要硬挤一个
- [ ] 测试 Worker 的非生产分支构建命令 = `wrangler deploy`（不是 upload）
- [ ] 部署后同时验证：临时预览 URL **和** 自定义域名
- [ ] 环境变量、Webhook 回调地址按环境分开配，别串
